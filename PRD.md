# PlateIQ — Product Requirements Document

**Project:** PlateIQ
**Type:** 8-Day Capstone Project
**Owners:** Victor Castillo & Juan Franco
**Date:** March 2026

---

## Problem

55% of American calories come from ultra-processed foods, yet the tools available to consumers focus almost exclusively on calorie counting. Existing nutrition apps miss the bigger picture: food quality, processing level, and ingredient transparency.

## Solution

PlateIQ is an AI-powered food wellness dashboard. Users snap a photo of their meal (or upload one), and the app uses Claude's Vision API to identify foods, estimate portions, and generate a visual wellness report — shifting the focus from calorie enumeration to food quality understanding.

---

## Users

| Segment | Description |
|---|---|
| Primary | Health-conscious consumers seeking deeper food insight beyond macro tracking |
| Secondary | Capstone judges evaluating technical execution and product quality |

---

## Core Value Props

1. **Instant nutrition breakdown** via photo — no manual logging
2. **NOVA processing-level transparency** — food classification on a green → red scale
3. **Personalized AI insights** — specific food swap suggestions generated by Claude

---

## Technical Stack

| Layer | Technology |
|---|---|
| Frontend | React + Vite |
| Data Viz | Recharts |
| Food AI | Claude Vision API |
| Nutrition Data | USDA FoodData Central API |
| Health Context | CDC PLACES API (P2) |
| Camera | Browser MediaDevices API |

---

## Data Flow

```
Image capture
  → Claude Vision analysis (structured JSON)
  → USDA nutrient lookup
  → Data merge
  → Dashboard render
  → Second Claude call → personalized wellness insights
  → Real-time recalculation on portion adjustment
```

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Claude Vision API failure during demo | Pre-cached demo meals as fallback |
| Camera access blocked | File upload alternative always available |
| USDA / CDC API rate limits | Local caching of API responses |

---

## Constraints

- Feature freeze after **Day 5**
- Days 6–8 reserved for polish and presentation prep
- No user authentication or data persistence required (MVP)
- No historical meal tracking in MVP scope
- Must function in offline/demo mode for live presentation
